01Solution Architecture

System Overview

The system is a geo-spatial, real-time simulation platform with three primary responsibilities: (1) rendering ~10,000+ HDB 3D building footprints with WebGL at 60fps, (2) running a stochastic demographic simulation engine server-side, and (3) persisting named simulation sessions with full replay capability.

Client (Browser)
deck.gl + Mapbox GL JS
3D tile rendering, ScenegraphLayer for HDB OBJ models, colour-mapped vacancy overlays
React 18 + Vite
Component tree, side panels, simulation controls, session management UI
Zustand State Store
Simulation state, year cursor for timeline replay, selected building context
WebSocket Client
Receives yearly tick events, move logs, aggregate stats in real time

FastAPI (Python)
REST endpoints: buildings, sessions CRUD, simulation controls. WebSocket hub for live ticks

Async background task: births, deaths, inter-flat moves per simulated year using NumPy vectorised ops

Offloads long-running or scheduled sim ticks; pub/sub for WebSocket fan-out

Simulation run state, WS message broker, session cache

02Technology Selection

Stack Rationale

3D Renderer
deck.gl
GPU-accelerated WebGL2 layer system purpose-built for geo data. ScenegraphLayer renders thousands of glTF/OBJ models with instancing. Handles Singapore-scale datasets at 60fps where raw Three.js would require manual instancing and frustum culling.
Base Map
Mapbox GL JS
Native deck.gl integration. Provides Singapore basemap context. The island boundary GeoJSON renders as a FillLayer with solid green fill at no extra cost. Alternative: MapLibre GL (OSS) to avoid Mapbox API fees.
Frontend
React 18 + Vite
Concurrent rendering for smooth UI during heavy WebGL draw calls. Vite's HMR is fastest in class for iteration speed. Zustand chosen over Redux — minimal boilerplate, fine-grained subscriptions, no provider wrapping.
API Server
FastAPI
Native async support is critical — WebSocket fan-out and Celery integration both require async I/O. Auto-generated OpenAPI docs speed up frontend integration. Python ecosystem gives instant access to NumPy for vectorised sim logic.
Database
PostgreSQL + PostGIS
PostGIS geom columns store building footprints; spatial queries identify nearest neighbours for moving-resident logic. JSONB columns efficiently store variable-length yearly event logs per simulation.
Task Queue
Celery + Redis
Simulation ticks can run 1–30s per year. Celery keeps API responsive by offloading CPU work. Redis double-serves as both Celery broker and WebSocket pub/sub channel, reducing infrastructure components.
Cloud
GCP Cloud Run
Serverless containers scale to zero between sessions — ideal for "medium traffic" SLAs where billed compute should match actual usage. Cloud SQL (Postgres) + Memorystore (Redis) complete the managed data tier.
Containers
Docker Compose
Single docker-compose up spins up frontend dev server, FastAPI, Celery worker, Postgres, and Redis locally — zero manual setup for reviewers running the MVP.

Alternatives Considered

LayerChosenAlternativesWhy Not
3D Engine deck.gl chosen Three.js, CesiumJS Three.js requires manual instancing for 10k+ models. CesiumJS is heavier and more complex to style; better for satellite/terrain imagery.
API FastAPI chosen Node/Express, Django, Go Fiber Express lacks native WebSocket+async task integration. Django is heavier for a prototyping context. Go Fiber is fast but Python's NumPy vectorisation wins for sim logic.
Database PostgreSQL chosen MongoDB, Supabase, SQLite MongoDB lacks native spatial indexing for neighbour queries. SQLite doesn't support concurrent writes from Celery workers. Supabase is Postgres under the hood — could be used if no-ops is preferred.
State Mgmt Zustand chosen Redux Toolkit, Jotai, React Query Redux is over-engineered for this scope. Jotai is fine but Zustand's devtools + middleware ecosystem is more mature. React Query handles server state but not WebSocket-pushed client state.
Deployment Cloud Run chosen GKE, App Engine Flex, Firebase GKE is cost-prohibitive at medium traffic (always-on nodes). App Engine Flex has ~1min cold start. Firebase Hosting covers frontend only. Cloud Run cold starts are <2s with min-instances=0.
03Implementation Strategy

Priority & Sequencing

User stories are ordered by the value delivered per unit of risk. The 3D map is the visual centrepiece — if it doesn't perform, no other feature matters. Simulation logic is the unique technical challenge. Session persistence and polish are valuable but can be iterated.

P1
3D Map Rendering — HDB Buildings + Island Boundary

Highest perceived value; validates the core tech bet (deck.gl + CityJSON pipeline). Must hit 60fps on mid-tier laptops with all ~12,000 HDB blocks visible. Island boundary GeoJSON layer added alongside as it's a trivial FillLayer addition. Risk: CityJSON → OBJ/glTF conversion pipeline complexity; addressed in first task.

P1
Building Info Panel — Click to Select

Direct dependency on P1 map work. Selecting a building and showing address + unit breakdown is a demo moment — evaluators will click immediately. Requires buildings table seeded from hdb.json. No extra infra needed.

P2
Simulation Engine — Assign, Tick, Pause

Core novel feature. Assign residents → run yearly ticks → WebSocket push to client. Colour-coded occupancy map updates per tick. This is the unique technical demonstration. Risk: Celery + WebSocket fan-out integration; prototype with in-process async first, then offload.

P2
Side Panel — Stats & Event Log

Low implementation cost once WS events flow. Year, total residents, deaths, move log. Adds significant demo polish with minimal extra work. Build alongside simulation engine.

P3
Session Persistence & Timeline Replay

High user value for the "restore earlier session" story. Requires DB schema for snapshots. The timeline scrubber is a UI-only concern once snapshots exist — it just seeks to a stored year state. Deferred to P3 as it requires more DB design.

P4
GCP Deployment — Cloud Run + Terraform

Important for deliverables but last in sequence — everything must work locally first. Terraform configs can be generated quickly once infra requirements are finalised. Cloud Run deploy from container image is straightforward.

05Cloud Architecture

GCP Deployment — Minimal Cost at Medium Traffic

The guiding principle: pay only when traffic arrives. Cloud Run's scale-to-zero model means the app costs ~$0 during off-hours while handling bursts of concurrent users without over-provisioning.

Compute

  • Cloud Run — API service min-instances=1 (warm standby)
  • Cloud Run — Celery worker min-instances=0
  • Cloud CDN + GCS 3D assets + frontend static
  • Firebase Hosting (alt.) frontend SPA fallback

Data

  • Cloud SQL (Postgres 15) db-f1-micro, shared CPU
  • Memorystore Redis M1 Basic, 1GB
  • GCS Standard bucket multi-regional, assets
  • Cloud SQL auto-backup 7-day retention

Networking

  • VPC Connector Cloud Run ↔ private SQL/Redis
  • Cloud Load Balancer HTTPS termination + CDN
  • Cloud Armor basic WAF, DDoS protection
  • Custom domain via Cloud DNS optional

Estimated Monthly Cost

  • Cloud Run (API) ~$15–30
  • Cloud SQL db-f1-micro ~$10
  • Memorystore Redis 1GB ~$35
  • GCS + CDN egress ~$5–15
  • Total ~$65–90/mo
Cost optimisation note: Redis Memorystore is the dominant cost at medium traffic. If budget is constrained, consider Upstash Redis (serverless, ~$0.2/100k commands) as a managed alternative that scales to near-zero. Cloud SQL can use the db-f1-micro shared-CPU tier which costs ~$7/month for this workload.

07Risks & Mitigations

Technical Risks

RiskLikelihoodMitigation
CityJSON geometry parsing complexity — some buildings may have malformed / LoD2-only geometry Medium Use cjio library's validation pass; fall back to extruded footprint polygon if solid geometry unavailable. Both render acceptably in deck.gl.
60fps performance with 12,000+ 3D instances Medium deck.gl's ScenegraphLayer uses GPU instancing — tested to 100k+ instances. Use simplified low-poly glTF model per building. Add LOD: show extruded polygon layer at zoom < 14.
WebSocket disconnection during long sim runs Low-Med Client reconnects automatically; server replays last N ticks from Redis stream on reconnect. Simulation state is always the DB truth, not the WebSocket.
Cloud Run cold starts interrupting WebSocket sessions Low Set min-instances=1 on the API service to eliminate cold starts for active sessions. Worker can still scale to zero.
Mapbox token cost / rate limits Low Swap to MapLibre GL JS (OSS drop-in) + OpenStreetMap tiles or Singapore's OneMap tiles, which are free for SG-based apps.
